Case Study 04

Enterprise RAG Pipeline with Tool Calling

Azure Databricks RAG HNSW Indexing Gemma 3 LangChain

Building a production-grade Retrieval-Augmented Generation pipeline that handles 10,000+ daily queries with sub-second latency, enabling intelligent document retrieval and structured tool calling for a leading NBFC.

The Problem

Why We Built This

Shriram Finance, one of India's largest NBFCs, manages an enormous repository of loan policies, compliance documents, product specifications, and procedural guidelines. Loan officers and customer service representatives spent 2-4 hours daily searching through PDFs, SharePoint folders, and legacy systems to answer customer queries accurately.

Pain Points

  • Information Silos: Critical policy documents scattered across 15+ different systems with no unified search
  • Response Latency: Average query resolution time of 45 minutes, leading to customer churn
  • Accuracy Issues: 23% of customer responses contained outdated or incorrect information
  • Training Overhead: New employees took 3+ months to become proficient in document navigation
  • Compliance Risk: No audit trail of what information was provided to customers

The leadership team needed a solution that could provide instantaneous, accurate answers grounded in official documents, with the flexibility to query structured data (loan balances, payment schedules) alongside unstructured content.

Architecture

The Solution Design

We architected a hybrid RAG system combining dense vector retrieval with structured tool calling capabilities. The key insight was that not all queries are equal—some require semantic understanding of policy documents, while others need precise database lookups.

Why Azure Databricks?

We chose Azure Databricks as our ML platform for three critical reasons:

  • Unified Vector Store: Databricks Vector Search provides native HNSW indexing with automatic index management, eliminating the need for separate vector database infrastructure
  • MLflow Integration: End-to-end experiment tracking, model versioning, and deployment pipelines in one ecosystem
  • Enterprise Security: Native integration with Azure AD, row-level security, and compliance with RBI data localization requirements

Implementation

Technical Deep-Dive

1. Document Ingestion Pipeline

The ingestion pipeline processes 50,000+ documents daily using a multi-stage approach: 

from databricks.sdk import WorkspaceClient
from langchain.text_splitter import RecursiveCharacterTextSplitter
from sentence_transformers import SentenceTransformer

class DocumentIngestionPipeline:
    def __init__(self):
        self.embedder = SentenceTransformer('all-MiniLM-L6-v2')
        self.splitter = RecursiveCharacterTextSplitter(
            chunk_size=512,
            chunk_overlap=50,
            separators=["\n\n", "\n", ". ", " "]
        )

    def process_document(self, doc_path: str):
        # Extract text based on file type
        text = self.extract_text(doc_path)

        # Chunk with metadata preservation
        chunks = self.splitter.split_text(text)

        # Generate embeddings in batches
        embeddings = self.embedder.encode(chunks, batch_size=32)

        # Upsert to Databricks Vector Search
        self.vector_store.upsert(
            chunks=chunks,
            embeddings=embeddings,
            metadata={
                'source': doc_path,
                'timestamp': datetime.now(),
                'department': self.classify_dept(doc_path)
            }
        )

3. Tool Calling with Gemma 3

The breakthrough came from implementing structured tool calling. Gemma 3's instruction following capabilities allowed us to define precise schemas: 

TOOL_SCHEMAS = {
    "get_loan_balance": {
        "description": "Retrieve current loan balance and payment status",
        "parameters": {
            "loan_id": {"type": "string", "pattern": r"^LN\d{8}$"},
            "customer_id": {"type": "string"}
        },
        "required": ["loan_id"]
    },
    "calculate_emi": {
        "description": "Calculate EMI for new loan or restructuring",
        "parameters": {
            "principal": {"type": "number", "minimum": 10000},
            "tenure_months": {"type": "integer", "minimum": 6, "maximum": 84},
            "interest_rate": {"type": "number"}
        },
        "required": ["principal", "tenure_months"]
    },
    "check_eligibility": {
        "description": "Check loan eligibility based on customer profile",
        "parameters": {
            "customer_id": {"type": "string"},
            "loan_type": {"enum": ["personal", "business", "gold"]}
        }
    }
}

def execute_tool_call(tool_name: str, params: dict) -> dict:
    if tool_name == "get_loan_balance":
        return db.sql(f"""
            SELECT loan_id, principal_amt, outstanding,
                   next_due_date, days_past_due
            FROM loans.loan_master
            WHERE loan_id = '{params['loan_id']}'
        """).to_dict()

    elif tool_name == "calculate_emi":
        # EMI = P * r * (1+r)^n / ((1+r)^n - 1)
        P = params['principal']
        r = params['interest_rate'] / 12 / 100
        n = params['tenure_months']
        emi = P * r * (1+r)**n / ((1+r)**n - 1)
        return {"emi": round(emi, 2), "total_payment": round(emi*n, 2)}

4. Query Routing Logic

The hybrid approach required intelligent query classification: 

from typing import Literal

class QueryRouter:
    def classify(self, query: str) -> Literal["rag", "tool", "hybrid"]:
        # Keywords suggesting structured data need
        tool_indicators = [
            "balance", "emi", "due date", "amount",
            "status", "eligible", "calculate", "my"
        ]

        # Keywords suggesting policy/procedural info
        rag_indicators = [
            "policy", "procedure", "how to", "what is",
            "guidelines", "rules", "can I", "eligible"
        ]

        query_lower = query.lower()
        has_tool = any(kw in query_lower for kw in tool_indicators)
        has_rag = any(kw in query_lower for kw in rag_indicators)

        if has_tool and has_rag:
            return "hybrid"
        elif has_tool:
            return "tool"
        else:
            return "rag"

Key Learnings

  • Chunk Size Matters: 512 tokens with 50-token overlap provided optimal retrieval for policy documents
  • Hybrid Approach is Essential: 40% of queries needed both RAG and tool calling
  • HNSW Tuning is Non-Trivia: ef_search=128 was the sweet spot for our recall/latency requirements
  • Tool Schemas Need Iteration: We refined our schemas 4 times based on error patterns
